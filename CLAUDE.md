# 面接無双 — CLAUDE.md

## プロジェクト概要
就活生向けの面接練習Webアプリ「面接無双」。日本人面接官との対話形式で一次〜最終面接を練習でき、回答に対してルールベースの5観点添削とコーチチャットでフィードバックを返す。パスワードゲート付きで友人限定に公開中。

## 技術スタック
- **単一HTMLファイル**の静的サイト（`index.html`）。ビルド不要・外部通信なし・フレームワーク不使用（素のJS/CSS）
- 画像はすべてbase64埋め込み（面接官の写真6枚、ヒーロー写真等）
- データ永続化は`localStorage`のみ（`mm_gate_v1`＝パスワードゲート通過フラグ、`mensetsu_history_v1`＝面接履歴、最大50セッション）
- ホスティング: GitHub Pages（リポジトリ直下の`index.html`をそのまま配信）

## ディレクトリ構成の要点
```
mensetsu-musou/
├── index.html    # アプリ本体（このファイルが全て。単一ファイルで完結。常に最新Volのコピー）
├── versions/     # 完了した各フェーズのスナップショット（Vol1.0〜、詳細はVERSIONS.md）

├── README.md     # ユーザー向け概要・クレジット
└── CLAUDE.md     # このファイル
```
- 実際の編集作業は**このリポジトリの`index.html`を直接編集**する（セッション固有のscratchpadは会話終了で消えるため使わない）
- 意味のあるまとまり（バグ修正・機能追加など）が完了したら、`index.html`を`versions/VolX.Y_短い説明.html`としてコピー保存してからコミット。詳細はVERSIONS.md参照
- リサーチノート（企業別質問データの出典等）は各セッションのscratchpad側にあり、このリポジトリには含めていない

## 現在の進捗（Vol 7.0: 完了・公開済み）
評価レポートの質を上げる改修（旧称「v14編集」）。対応項目:

**本スコープ（優先度高）**
1. スコア説明力バグ修正 — `analyze()`のscoreが全findingsから計算される一方、画面表示は減点理由を最大3件に切り詰めていて不一致だった。重み付け（構造的欠陥=1点／文体レベルの軽微な癖=0.5点）＋「ほか+N件」表示で解消
2. `DEEP_TYPES`拡張 — 数字・構成の評価軸が`motivation`/`industry`タイプの質問に適用されていなかった問題を修正
3. 5観点メーターの小標本(n)配慮 — 対象問題数が少ない軸に「参考」ラベルを表示し、%の過大解釈を防止
4. 自由質問欄のEnterキー問題修正 — `<input type="text">`だったため改行できず即送信されていたのを`<textarea>`化。Shift+Enterで改行、Enterで送信
5. `routeFreeText`/コーチ返答の分岐拡張 — ルールベースのまま「短く」「他社」「自己紹介」「緊張」等の分岐を追加

**軽微2項目（本スコープに含めて一緒に対応）**
- コーチチップ（なぜこの評価？等）に既読の見た目をつけて連打時の重複感を軽減
- 総評のダメ出しフォールバックを配列順ではなく重要度順（`BAD_PRIORITY`）でソート

## Vol 8.0: 添削レポートの根拠強化（2026-07-14 実装・ブラウザ実機検証完了）

**背景**: 現在の`analyze()`（index.html ~2022行目）はすべて正規表現・キーワードマッチによる表層パターン判定であり、意味を理解して採点しているわけではない。この前提（静的サイト・バックエンド無しでLLM採点はできない）は維持したまま、「指摘の根拠が見える・納得感がある」方向に改善する。

**確定仕様**:

1. **原文引用型フィードバック**: 指摘（finding）を出すたびに、該当パターンにマッチした実際の回答文（一文）を引用して表示する。「あなたの回答:『〇〇』→ここに数字がありません」のような形式。判断ロジック自体は既存ルールベースを踏襲（4番の抽象語・6番の数字検出は仕様変更あり）。
2. **原文表示のUI**: 引用は常時インライン表示ではなく、「あなたの文章」というボタン/リンクをクリックしたときだけ展開表示する（折りたたみ式）。
3. **分量評価を質問タイプ別に**: 現状は`item.t !== "reverse"`のみで一律80字未満/550字超の閾値を使っているが、質問タイプに応じて基準を変える。自己紹介のような性質の質問は短くてよく、短さで減点しない。エピソード系（`DEEP_TYPES`に該当する質問）は現状通りある程度の分量を求める。→ 質問タイプ→長さ下限/上限のテーブルを新設する。
4. **長さと構成のトレードオフ**: 長すぎる場合（`too_long`）でも、起承転結にあたる構成（`structure`のfinding、課題→行動→結果の要素）がしっかりしていれば、長さの減点を緩和・相殺する。単純な文字数だけで機械的に減点しない。
5. **主体性判定を柔軟に**: 「私は」を使う方が望ましいが一律の減点ルールにはしない。「私たち」始まりでも、後続で個人の判断・行動を示す表現（担当した／主導した／決めた等）があれば主体性ありと判定し、`we_only`扱いにしない。文脈に応じた柔軟な判定にする。
6. **抽象語の評価変更**: 「様々／いろいろ」等の単純なカウントに加え、①固有名詞・数字・独自の言い回しなど具体的な表現を積極的に評価（goodな finding として追加）、②同じ語彙・言い回しを回答内で繰り返し使っている場合は減点（新しい bad finding）。
7. **文単位のタグ付け評価**: 回答全体への一括判定に加えて、文ごとに簡易タグ（結論◎／数字△など）を出せるようにする。
8. **数字検出の精度向上**: 現状の`hasNumber`は文中に数字が1つでもあれば真判定になる緩い条件。日付や質問文の引用等の誤検出を減らし、成果指標らしい数字（人数・期間・割合・順位等）かどうかをもう少し絞り込む。

**実装状況（完了）**: Fable 5エージェントが実装。主な変更点は `LEN_RULES`（質問タイプ別分量閾値、introは30/450に緩和）、`too_long`の構成トレードオフ（`data.soft`で重み0.5→0.25）、`OWN_VERB_RE`（主導/担当/任され等で「私たち」でもwe_only抑制）、`repetition`（同一抽象語2回以上で減点）と`concrete_expression`（固有名詞・独自表現を加点）の新設・`vague`の役割分離、`NUM_UNIT_RE`（単位つき数字のみ意味のある数字扱い）、`meta.sentTags`（文単位タグ）、表示側の`fbLiHtml()`で「あなたの文章」引用トグル＋「文ごとの評価を見る」折りたたみ。Node単体テスト36件パス＋Claude Browserで7問フロー実機検証済み（原文引用トグル・文タグ・主体性緩和・数字検出・concrete加点すべて期待通り、コンソールエラーなし）。`versions/Vol8.0_evidence-based-feedback.html`へスナップショット済み。次の新規作業は**Vol 9.0**として着手する。

## 直近の決定事項
- **本物のClaude API連携（自由質問への真の対話応答）は別タスクとして切り出す**。理由: このアプリは静的HTML＋GitHub Pagesでバックエンドがなく、APIキーをHTMLに埋め込むとソースから誰でも読めて漏洩する。実現するには製品版でキーを守るバックエンド（プロキシ）が必要になるため、今回のスコープからは意図的に外している
- `company_specific`判定のゲーム化しやすさ（社名を1回入れるだけで加点される問題）は今回は優先度低として保留
- 作業は5時間の利用上限を考慮し、可能な限り細かい単位でgitにコミット/pushして進捗を残す方針

## 実行・テストコマンド
このプロジェクトにはビルド・パッケージマネージャは無い。動作確認は以下の手順。

```bash
# ローカルプレビュー（任意のポートで配信するだけ）
python3 -m http.server 8642 --directory .
# → http://localhost:8642/index.html をブラウザで開く

# JS構文チェック（<script>内を抜き出してNodeでチェック）
python3 -c "
import re
h = open('index.html', encoding='utf-8').read()
scripts = re.findall(r'<script>(.*?)</script>', h, re.S)
open('/tmp/_main_script.js','w',encoding='utf-8').write(max(scripts, key=len))
"
node --check /tmp/_main_script.js
```

デプロイは`git add index.html && git commit && git push`のみ（GitHub Pagesが自動反映、1〜2分）。
